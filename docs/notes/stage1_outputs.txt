● The merged model is still partially quantized — merge_and_unload() didn't dequantize back to full precision. Here's what actually happened:

  base:   338 tensors  |  1.544B params  |  all bfloat16       → 3.09 GB
  merged: 1318 tensors |  0.909B params  |  mixed dtypes        → 1.14 GB

  The merged file has three kinds of tensors:

  ┌──────────┬───────┬────────┬──────────────────────────────────────────────────────┐
  │  Dtype   │ Count │ Params │                         What                         │
  ├──────────┼───────┼────────┼──────────────────────────────────────────────────────┤
  │ bfloat16 │ 142   │ 233M   │ Embeddings, layer norms, biases — never quantized    │
  ├──────────┼───────┼────────┼──────────────────────────────────────────────────────┤
  │ uint8    │ 588   │ 676M   │ All 196 weight matrices still in 4-bit NF4           │
  ├──────────┼───────┼────────┼──────────────────────────────────────────────────────┤
  │ float32  │ 588   │ 0.1M   │ Per-layer quantization metadata (absmax, quant maps) │
  └──────────┴───────┴────────┴──────────────────────────────────────────────────────┘

  The 676M weight values stored in uint8 use 2 NF4 values packed per byte — that's why they're ~4× smaller than they'd be in bf16. merge_and_unload() merged the LoRA deltas into the quantized base weights but left them in 4-bit format. The rounding error warning at merge time was
  the hint.

  Does this matter? For Stage 2 training it's fine — Stage 2 re-quantizes the model anyway when loading with BitsAndBytesConfig. For inference it's also fine — bitsandbytes loads it transparently. The only thing you can't do is load it as a plain HuggingFace model without
  bitsandbytes.

  If you want a true full-precision merged model the fix is to load the base in fp16 before merging:

  # Load base in fp16 (no BitsAndBytesConfig), apply adapter, merge
  model = AutoModelForCausalLM.from_pretrained(str(base), dtype=torch.float16)
  model = PeftModel.from_pretrained(model, str(adapter))
  merged = model.merge_and_unload()   # now truly fp16, ~3 GB

  That would produce a ~3 GB fp16 file with no bitsandbytes dependency. Worth doing if you plan to use the merged model for anything outside this pipeline (e.g. exporting to GGUF for llama.cpp).


❯ Why are the number of tensors and params different between base and merged?
  Create a script that can re-run the merge with the base loaded in fp16 and save it.

● Tensor count (338 → 1318): In the base file, each weight matrix is one bf16 tensor. When bitsandbytes quantizes a weight it splits it into 6 tensors: the packed uint8 values plus 5 quantization state tensors (absmax, nested_absmax, nested_quant_map, quant_map, quant_state). So
  196 weights × 6 = 1,176 quantization tensors + 142 unquantized (embeddings, norms, biases) = 1,318.

  Param count (1.544B → 0.909B): NF4 packs 2 values per byte, so numel() on a uint8 tensor reports half the actual parameter count. The 676M uint8 elements actually represent ~1.35B weight values. Add the 233M bf16 params and you recover roughly the expected 1.58B — close to the
  base's 1.544B (small discrepancy from quantization block-size padding).
